# Loop Compilation

The loop document is generated by the skill. It is a declarative execution contract, not a scheduler state database.

## Required content

A generated loop should contain:

1. **Outcome** — one precise statement of the intended result.
2. **Acceptance demo** — the smallest observable flow proving value.
3. **Scope and non-goals** — what may and may not change.
4. **Repository evidence** — current architecture, relevant files, tests, constraints, and assumptions.
5. **Authority** — local merge, push, PR, deployment, external system, and destructive-action boundaries.
6. **Consumption tier & budget** — the chosen tier (light / standard / aggressive) with one line of reasoning and a rough cost envelope, plus hard caps: max concurrent agents, max total agents, max recursion depth, max fix-waves per gate (default 3), and any wall-clock or token ceiling. See [PARALLELISM.md](PARALLELISM.md) for the tiers.
7. **Exit states** — what `success`, `budget-exhausted`, and `blocked-handoff` each mean for this loop and what each returns.
8. **Oracles** — how every acceptance criterion will be verified.
9. **Contracts** — interfaces, data shapes, behavior, and ownership boundaries that parallel workers must respect.
10. **Task graph** — dependencies, waves, and integration points.
11. **Native topology** — which work uses subagents, teams, background sessions, or worktrees.
12. **Review topology** — fresh reviewers and deterministic gates.
13. **Integration policy** — how accepted changes reach the integration head and target branch.
14. **Stop/escalation rules** — what triggers repair, repartition, budget-exhaustion, or a human decision.
15. **Cleanup definition** — agents, worktrees, branches, runtime resources, and code debris.

See [EXAMPLE_LOOP.md](EXAMPLE_LOOP.md) for a fully worked instance of all of the above, compiled from a rough requirement.

Scale the document to the tier. At the **light** tier, keep every section present (so the artifact stays uniform and resumable) but let items 8–15 collapse to one or two lines each — do not over-document a one-file change. At **aggressive**, expand topology, variants, and review lanes. The Budget and the three exit states are mandatory at every tier, however small.

## Do not include

Do not create:

- a live agent registry that duplicates the harness;
- lease or heartbeat records;
- a custom queue or scheduler;
- a daemon or controller command language;
- per-turn transcript copies;
- JSON state merely to make the process appear engineered.

The harness owns live state. The loop document remains stable enough to hand to another compatible harness or resume after a context change.

## Task-specific loop shapes

These are reusable skeletons. From the rough requirement, infer which shape fits and start from its skeleton and suggested default tier, then adjust. They are starting points, not categories the user must name.

A shape's default tier is a starting point, not a hard assignment. When the shape default and the requirement's size disagree (e.g. a *small, well-scoped* focused feature), the recommend-by-signal table in [PARALLELISM.md](PARALLELISM.md) is the tiebreaker — there, a clear-boundary localized change drops to **light** even though the focused-feature shape defaults to standard.

### Focused feature

Use one fleet loop with discovery, contract freeze, implementation waves, independent review, integration, and cleanup. *Default tier: standard.*

### Complete product or large product increment

Use three nested levels:

- **Product loop:** chooses the next valuable vertical slice and validates product direction.
- **Feature loop:** closes one end-to-end user journey.
- **Coding loop:** implements and verifies bounded technical tasks.

Parallelize within a vertical slice, but keep the outer product loop responsible for sequencing value. Give the product loop a budget across all three levels so nested fan-out cannot compound past the ceiling. *Default tier: aggressive (it is the explicit cost-secondary case).*

### Bug campaign

Fan out independent reproduction and root-cause hypotheses, select evidence-backed causes, then run bounded fix and regression waves. *Default tier: standard, dropping to light once the cause is confirmed.*

### Migration or refactor

Inventory the surface in parallel, freeze compatibility contracts, transform independent shards, continuously run compatibility/regression gates, and integrate in small waves. *Default tier: standard.*

### Broad repetitive change

Partition by independent package/file groups, assign one writer per shard, use shared deterministic validation, then run a cross-shard integration review. *Default tier: light per shard (the work is mechanical and the deterministic oracle is the real gate).*

## Portable loop file

When a persistent artifact is useful, create `.ai/loops/<slug>.md` or follow the repository's established convention. It should be generated after intake and repository inspection. It is not a form for the user to fill out.
