# Phase 2.3 — Graph Streaming Pipeline

> Design document for replacing `graph.invoke()` with `graph.stream()`,
> adding an OrchestrationService, bridging node lifecycle events to EventBus,
> and connecting LLM streaming chunks.

## 1. Problem Statement

`graph.invoke()` is fire-and-forget. The caller waits for the entire graph
execution (boss → manager → employee → summary) to complete before receiving
any result. During execution:

- EventBus subscribers cannot observe which node is currently active
- No `graph.node.entered` / `graph.node.exited` events exist
- `bossSummaryNode`'s streaming LLM chunks are discarded (`onChunk: () => {}`)
- There is no orchestration layer — tests call `graph.invoke()` directly

This blocks Phase 3+ UI work (PixiJS office scene, runtime shell) which needs
real-time visibility into agent execution progress.

## 3. Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Stream API | `graph.stream()` (not `streamEvents(v2)`) | Our nodes already emit LLM-level events; `streamEvents` would create duplicates and couple us to LangGraph's internal event format |
| Stream mode | `streamMode: 'updates'` | Yields `{ nodeName: partialState }` — gives node attribution |
| Node entered events | Emitted by nodes themselves | `graph.stream()` only signals completion; nodes know when they start |
| Node exited events | Emitted by OrchestrationService | Service receives stream updates with node names |
| onChunk connection | Yes, in Phase 2.3 | Small change (1 line + 1 factory), enables Phase 3 token-level UI |
| OrchestrationService scope | Thin wrapper (A+) | Nodes already manage lifecycle; service just bridges streaming to EventBus |
| UI / Runtime Shell | Out of scope | Phase 2.3 is pure core pipeline; UI is Phase 3 |

## 4. Architecture

### Event Flow Diagram

```
Caller
  │
  ▼
OrchestrationService.execute(entryMode, messages)
  │
  ├─ Assembles fullInput (threadId/companyId from runtimeCtx)
  ├─ Assembles config (thread_id + runtimeCtx)
  │
  ▼
graph.stream(fullInput, { ...config, streamMode: 'updates' })
  │
  │  yields: { boss: {...} }
  │  yields: { manager: {...} }
  │  yields: { employee: {...} }
  │  yields: { boss_summary: {...} }
  │
  ├──── per yield ──→ eventBus.emit(graph.node.exited)
  │
  └──── returns final AicsGraphState

Inside each node (concurrent with stream):
  │
  ├─ Start: eventBus.emit(graph.node.entered)           ← NEW
  │
  ├─ LLM: recordedLlmCall / recordedLlmStream
  │   ├─ llm.call.started                               (existing)
  │   ├─ llm.stream.chunk (boss_summary only)            ← NEW
  │   └─ llm.call.completed                             (existing)
  │
  └─ Domain events:
      ├─ employee.state.changed                          (existing)
      ├─ task.state.changed                              (existing)
      ├─ task.assignment.changed                         (existing)
      └─ meeting.state.changed                           (existing)
```

### Why Not `streamEvents(v2)`?

Our architecture already has fine-grained events at the node level through
`recordedLlmCall` / `recordedLlmStream` → EventBus. Using `streamEvents(v2)`
would:

1. **Duplicate events** — LangGraph's `on_llm_start` overlaps our `llm.call.started`
2. **Couple to internals** — LangGraph's event format may change between versions
3. **Flood EventBus** — `streamEvents` emits channel updates, edge routing, etc.

`graph.stream()` fills the one gap we have: **node-level progress**.

## 6. Component Design

### 6.1 OrchestrationService

```typescript
// packages/core/src/services/orchestration-service.ts

import type { CompiledStateGraph } from '@langchain/langgraph';
import type { RuntimeContext } from '../runtime/runtime-context.js';
import type { AicsGraphState } from '../graph/state.js';
import type { BaseMessage } from '@langchain/core/messages';
import { graphNodeExited } from '../events/event-factories.js';

export class OrchestrationService {
  constructor(
    private graph: CompiledStateGraph,
    private runtimeCtx: RuntimeContext,
  ) {}

  async execute(input: {
    entryMode: AicsGraphState['entryMode'];
    messages: BaseMessage[];
  }): Promise<AicsGraphState> {
    const fullInput = {
      threadId: this.runtimeCtx.threadId,
      companyId: this.runtimeCtx.companyId,
      ...input,
    };
    const config = {
      configurable: {
        thread_id: this.runtimeCtx.threadId,
        runtimeCtx: this.runtimeCtx,
      },
    };

    let finalState: Partial<AicsGraphState> = { ...fullInput };

    for await (const update of this.graph.stream(fullInput, {
      ...config,
      streamMode: 'updates' as const,
    })) {
      for (const [nodeName, nodeOutput] of Object.entries(update)) {
        this.runtimeCtx.eventBus.emit(
          graphNodeExited(
            this.runtimeCtx.companyId,
            this.runtimeCtx.threadId,
            nodeName,
          ),
        );
        // Merge node output into accumulated state
        finalState = { ...finalState, ...(nodeOutput as Partial<AicsGraphState>) };
      }
    }

    return finalState as AicsGraphState;
  }
}
```

Design notes:
- `execute()` returns `Promise<AicsGraphState>` — same shape as `graph.invoke()`
- Callers can also subscribe to EventBus for real-time updates
- Phase 3 can add `executeStream()` returning `AsyncIterable<GraphStreamEvent>` if UI needs direct stream consumption

## 7. Backward Compatibility

`graph.invoke()` calls in existing tests continue to work. OrchestrationService
is **additive** — it does not replace `graph.invoke()`. Tests can choose either:

- `graph.invoke(...)` — for tests that don't care about streaming events
- `service.execute(...)` — for tests validating the streaming pipeline

The `graph.node.entered` events added to nodes will fire during `graph.invoke()`
too (they use the same EventBus), which is harmless — existing tests don't
assert against unknown event types.

## 11. Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `graph.stream()` update format differs from expectation | Medium | Task 1 validates format with a minimal test before building on it |
| `streamMode: 'updates'` type mismatch | Low | LangGraph 1.2.x TypeScript types should cover this |
| Node entered events fire twice in employee loop | Medium | Employee node's entered event fires per iteration — document as expected behavior |
| Existing tests break from new events | Very Low | Tests don't assert on unknown event types |

## 12. Success Criteria

Phase 2.3 is complete when:

1. OrchestrationService wraps `graph.stream()` with `streamMode: 'updates'`
2. `graph.node.entered` events emitted by all 5 agent nodes
3. `graph.node.exited` events emitted by OrchestrationService for each node completion
4. `llm.stream.chunk` events emitted during boss_summary streaming
5. Integration tests validate event sequences
6. All tests pass (100+)
7. Existing `graph.invoke()` tests unaffected
8. Build and typecheck pass
