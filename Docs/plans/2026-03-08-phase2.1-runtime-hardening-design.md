## 2. Track A: Streaming + Cost Tracking

### A1. Adapter Retry Logic

Add retry wrapper to both `AnthropicAdapter` and `OpenAiAdapter`.

```typescript
interface RetryConfig {
  maxRetries: number;       // default: 3
  baseDelayMs: number;      // default: 1000
  maxDelayMs: number;       // default: 30000
}

async function withRetry<T>(
  fn: () => Promise<T>,
  config: RetryConfig,
  isRetryable: (error: unknown) => boolean,
): Promise<T>;
```

Retry conditions:
- HTTP 429 (rate limit): retry up to `maxRetries`, exponential backoff with jitter
- HTTP 5xx (server error): retry up to `maxRetries`, exponential backoff
- HTTP 4xx (client error, not 429): do NOT retry, throw immediately
- Network error / timeout: retry up to `maxRetries`

Retry logic lives in a shared `llm/retry.ts` utility, consumed by both adapters. Not in graph nodes.

### A3. Gateway Streaming Interface

Extend `LlmGateway` with a streaming method. Existing `chat()` is preserved.

```typescript
interface LlmStreamChunk {
  content?: string;           // text delta (may be empty on non-content chunks)
  toolCalls?: ToolCallResult[]; // tool call deltas (streaming tool use)
  usage?: LlmUsage;           // only present on the final chunk
  done: boolean;              // true on the last chunk
}

interface LlmGateway {
  chat(request: LlmRequest): Promise<LlmResponse>;
  chatStream(request: LlmRequest): AsyncIterable<LlmStreamChunk>;
}
```

Both adapters implement `chatStream()`:
- **AnthropicAdapter**: uses `client.messages.stream()`, maps `content_block_delta` events
- **OpenAiAdapter**: uses `client.chat.completions.create({ stream: true })`, maps `choices[0].delta`

`usage` is populated only on the final chunk (Anthropic: `message_delta` event with `usage`; OpenAI: `usage` field on last chunk when `stream_options.include_usage` is set).

Retry logic from A1 applies to `chatStream()` — retries the entire stream on connection-level failures, but NOT mid-stream. Once tokens start flowing, a mid-stream failure is non-retryable (throw `LlmError` with `recoverable: false`).

### A4. graph.stream() Replaces graph.invoke()

Change the primary graph execution entry point:

```typescript
// Before (Phase 2.0)
const result = await graph.invoke(input, config);

// After (Phase 2.1)
for await (const event of graph.stream(input, config)) {
  // event contains: node name, state updates, stream mode metadata
}
```

**Critical design decision: graph.stream() events are a separate channel from EventBus.**

| Channel | Content | Consumers |
|---------|---------|-----------|
| `EventBus` | Domain events (state transitions, usage records) | Audit, scene renderer, cost tracker |
| `graph.stream()` yield | Graph execution events + boss token chunks | UI real-time display layer |

Phase 2.1 keeps these two channels independent. No `RuntimeOutputStream` merge layer — deferred to Phase 2.2.

The `buildAicsGraph()` return type changes to expose both `invoke()` (kept for testing convenience) and `stream()` as the primary execution method.

### A5. Boss Token Streaming + Tee Pattern

Only `bossSummaryNode` consumes `chatStream()`. All other nodes continue using `chat()`.

The tee pattern solves the dual-consumer problem:
- **UI needs**: per-chunk forwarding for real-time display
- **Graph state needs**: collected full content for `messages` + `task_runs.output_json`

```typescript
interface TeeResult {
  fullContent: string;
  toolCalls: ToolCallResult[];
  usage: LlmUsage;
}

async function teeStream(
  stream: AsyncIterable<LlmStreamChunk>,
  onChunk: (chunk: LlmStreamChunk) => void,
): Promise<TeeResult> {
  let fullContent = '';
  let usage: LlmUsage = { inputTokens: 0, outputTokens: 0 };
  const toolCalls: ToolCallResult[] = [];

  for await (const chunk of stream) {
    onChunk(chunk);                          // side effect: forward to UI via graph.stream() yield
    if (chunk.content) fullContent += chunk.content;
    if (chunk.toolCalls) toolCalls.push(...chunk.toolCalls);
    if (chunk.usage) usage = chunk.usage;    // last chunk carries usage
  }

  return { fullContent, toolCalls, usage };
}
```

`onChunk` callback emits chunks through LangGraph's streaming mechanism (node yield), not through EventBus. EventBus receives only the final domain events (`llm.call.completed`, `llm.usage.recorded`).

### A6. Cost Tracking

#### New table: `llm_calls`

```sql
-- migration: 005_llm_calls.sql
CREATE TABLE IF NOT EXISTS llm_calls (
  llm_call_id   TEXT PRIMARY KEY,
  thread_id     TEXT REFERENCES graph_threads(thread_id) ON DELETE SET NULL,
  task_run_id   TEXT REFERENCES task_runs(task_run_id) ON DELETE SET NULL,
  node_name     TEXT NOT NULL,              -- 'boss' | 'manager' | 'employee' | 'meeting_participant' | ...
  provider      TEXT NOT NULL,              -- 'anthropic' | 'openai'
  model         TEXT NOT NULL,              -- 'claude-sonnet-4-20250514'
  input_tokens  INTEGER NOT NULL,
  output_tokens INTEGER NOT NULL,
  usage_raw_json TEXT,                      -- provider's full usage object, stored as-is
  response_json  TEXT,                      -- nullable; reserved for future deterministic replay (Phase 3+)
  latency_ms    INTEGER,                    -- end-to-end call duration
  error_code    TEXT,                       -- non-null on failure
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_llm_calls_thread ON llm_calls(thread_id);
CREATE INDEX idx_llm_calls_task_run ON llm_calls(task_run_id);
```

Design decisions:
- `input_tokens` / `output_tokens` are structured columns for fast querying
- `usage_raw_json` preserves provider-specific fields (Anthropic cache tokens, OpenAI reasoning tokens, etc.)
- `response_json` is nullable, not populated in Phase 2.1 — reserved for deterministic replay
- No price estimation — pricing is an external fact; token counts + model name are sufficient for users to compute cost
- Per-task and per-thread aggregation is derived from this atomic table via SQL, not stored redundantly

#### New EventFamily types

```typescript
// Added to shared-types EventFamily
| 'llm.call.started'       // payload: { llmCallId, nodeNname, provider, model, threadId }
| 'llm.call.completed'     // payload: { llmCallId, nodeName, latencyMs, inputTokens, outputTokens }
| 'llm.usage.recorded'     // payload: { llmCallId, threadId, taskRunId?, provider, model, inputTokens, outputTokens }
```

Event emission flow:
1. Before LLM call → emit `llm.call.started`
2. After LLM call (or stream completion) → emit `llm.call.completed`
3. After writing `llm_calls` row → emit `llm.usage.recorded`

All three events flow through EventBus (domain events), not graph.stream().

#### Integration into nodes

Each node that calls LLM wraps the call with timing + recording:

```typescript
async function recordedLlmCall(
  ctx: RuntimeContext,
  request: LlmRequest,
  meta: { nodeName: string; taskRunId?: string },
): Promise<LlmResponse> {
  const llmCallId = generateId();
  const startedAt = Date.now();

  ctx.eventBus.emit(llmCallStarted(ctx.companyId, llmCallId, meta.nodeName, ...));

  try {
    const response = await ctx.llmGateway.chat(request);
    const latencyMs = Date.now() - startedAt;

    await ctx.repos.llmCalls.create({
      llmCallId,
      threadId: ctx.threadId,
      taskRunId: meta.taskRunId ?? null,
      nodeName: meta.nodeName,
      provider: resolvedModel.provider,
      model: resolvedModel.model,
      inputTokens: response.usage.inputTokens,
      outputTokens: response.usage.outputTokens,
      usageRawJson: JSON.stringify(response.usage),
      responseJson: null,  // not populated in Phase 2.1
      latencyMs,
      errorCode: null,
      createdAt: new Date().toISOString(),
    });

    ctx.eventBus.emit(llmCallCompleted(ctx.companyId, llmCallId, ...));
    ctx.eventBus.emit(llmUsageRecorded(ctx.companyId, llmCallId, ...));

    return response;
  } catch (error) {
    // Record failed call with error_code
    await ctx.repos.llmCalls.create({ ...meta, errorCode: error.code, ... });
    throw error;
  }
}
```

A streaming variant `recordedLlmStream()` uses `teeStream()` internally and emits the same events after stream completion.

### B2. ExecutionTraceService

A pure-read service that assembles a cross-table execution trace. Designed for two future consumers: debug UI and scene renderer.

```typescript
interface ExecutionTrace {
  thread: GraphThreadRow;
  taskRuns: TaskRunRow[];              // ordered by started_at
  handoffs: HandoffEventRow[];         // ordered by created_at
  llmCalls: LlmCallRow[];             // ordered by created_at (Phase 2.1 new table)
  checkpoints: GraphCheckpointRow[];   // ordered by checkpoint_seq
  events: RuntimeEventRow[];           // ordered by created_at — the state transition timeline
}

interface ExecutionTraceService {
  /** Full execution trace for a thread */
  getTrace(threadId: string): Promise<ExecutionTrace>;

  /** State snapshot at a specific checkpoint */
  getStateAt(threadId: string, checkpointSeq: number): Promise<AicsGraphState>;

  /** List threads for a company */
  listThreads(companyId: string, opts?: { limit?: number; status?: string }): Promise<GraphThreadRow[]>;
}
```

Implementation queries across repositories — no new tables, only reads.

Design notes for future scene renderer consumption:
- All arrays are time-ordered — renderer can iterate linearly to reconstruct timeline
- `events` array contains `employee.state.changed` entries with `prev`/`next` and `created_at` — sufficient to drive animation state machines
- `llmCalls` with `latency_ms` enables realistic timing in replay (not just sequence, but duration)
- `checkpoints` with `state_json` enables "jump to point in time" for scrubbing

## 9. Design Decisions Log

| Decision | Rationale |
|----------|-----------|
| Two independent tracks (A/B) | No cross-dependencies; enables parallel development |
| Retry in adapter layer, not nodes | Consistent behavior; nodes don't need retry awareness |
| `chatStream()` as new method, `chat()` preserved | Backward compatible; nodes opt in to streaming |
| Tee pattern for boss streaming | Solves dual-consumer (UI + state) without buffering delay |
| graph.stream() separate from EventBus | Infrastructure events vs domain events; different consumers, different semantics |
| `llm_calls` as atomic cost data | Per-call granularity; per-task and per-thread are derived aggregations |
| `usage_raw_json` alongside structured columns | Query-friendly columns + future-proof raw storage (cache tokens, reasoning tokens) |
| `response_json` nullable, not populated | Schema reservation for deterministic replay without storage cost now |
| Smoke tests local-only, not CI | Open-source friendly; no secrets management burden |
| Adapter smoke before streaming work | Validates current sync adapters work before changing the interface |
| ExecutionTrace designed for scene renderer | Time-ordered arrays with duration data enable future animation replay |
| No RuntimeOutputStream in Phase 2.1 | Premature unification; let both channels prove their shape first |
