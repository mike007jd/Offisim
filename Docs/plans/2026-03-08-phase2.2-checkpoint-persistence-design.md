# Phase 2.2 — Checkpoint Persistence & Hardening

> Design document for replacing the in-memory DrizzleCheckpointSaver with
> LangGraph's official `SqliteSaver`, fixing the broken `getStateAt()` chain,
> and adding end-to-end persistence tests.

## 1. Problem Statement

Phase 2.1 shipped `DrizzleCheckpointSaver` — a class that claims Drizzle
persistence but stores everything in a `Map`. This means:

1. **Graph state is lost on process restart.** Desktop (the 1.0 reference
   environment) cannot resume interrupted workflows.
2. **`ExecutionTraceService.getStateAt()` is broken.** It reads from
   `CheckpointRepository` (our `graph_checkpoints` table), which is never
   written to by the in-memory saver.
3. **`meeting-flow` integration test lacks `llm_calls` coverage.** A gap
   identified during Phase 2.1 review.

## 2. Goals

| # | Goal | Metric |
|---|------|--------|
| G1 | Checkpoint state survives process restart | E2E test: execute graph, close DB, reopen, `getTuple` returns data |
| G2 | `getStateAt()` returns real graph channel state | Unit test with SqliteSaver |
| G3 | All existing tests pass (no regressions) | 88+ tests green |
| G4 | meeting-flow covers llm_calls recording | New assertion in integration test |

## 3. Key Decisions (Already Made)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Checkpoint backend | `@langchain/langgraph-checkpoint-sqlite` (official) | Eliminates reverse-engineering LangGraph's serialization format |
| DB instance sharing | SqliteSaver and Drizzle share the same `better-sqlite3` Database | Avoids WAL lock contention from dual connections |
| Package placement | `packages/core` (not `db-local`) | `db-local` is a pure schema package; all persistence implementations live in core |
| `getStateAt()` API | Change `(threadId, seq)` to `(threadId, checkpointId)` — no deprecated overload | No external consumers; one-step migration |
| `graph_checkpoints` table | Keep as-is, annotate `@reserved` for Phase 3 business snapshots | Table's `checkpoint_kind` enum (`meeting_turn`, `install_gate`, etc.) serves execution tracing, not LangGraph state |
| `graph.stream()` | Out of scope — deferred to Phase 2.3 | Isolate risk: don't change checkpoint pipeline and execution pipeline simultaneously |

## 4. Architecture Change

### Before (Phase 2.1)

```
buildAicsGraph()
  └─ createCheckpointSaver(db?: unknown)
       ├─ db provided  → DrizzleCheckpointSaver(db)  [Map in-memory!]
       └─ db omitted   → MemorySaver()

ExecutionTraceServiceImpl(repos)
  └─ getStateAt(threadId, seq)
       → repos.checkpoints.findBySeq()  ← graph_checkpoints table (never written)
```

### After (Phase 2.2)

```
buildAicsGraph()
  └─ createCheckpointSaver(db: Database)
       → new SqliteSaver(db)  [real SQLite persistence]
  └─ createMemoryCheckpointSaver()
       → new MemorySaver()    [testing]

ExecutionTraceServiceImpl(repos, checkpointSaver)
  └─ getStateAt(threadId, checkpointId)
       → checkpointSaver.getTuple(config)  ← SqliteSaver internal tables (real data)
```

### What SqliteSaver manages

SqliteSaver creates and manages its own internal tables (`checkpoints`,
`writes`) inside the shared SQLite file. These tables store:

- Full channel state (our `AicsGraphState`: messages, currentEmployee, etc.)
- Checkpoint metadata and parent pointers
- Pending writes for interrupted executions

Our `graph_checkpoints` table is **not** touched by SqliteSaver. The two
coexist in the same SQLite file with no name conflicts.

## 5. Component Changes

### 5.1 checkpoint-saver.ts (full rewrite)

```typescript
// packages/core/src/graph/checkpoint-saver.ts
import { MemorySaver, type BaseCheckpointSaver } from '@langchain/langgraph';
import { SqliteSaver } from '@langchain/langgraph-checkpoint-sqlite';
import type Database from 'better-sqlite3';

export function createCheckpointSaver(db: Database.Database): BaseCheckpointSaver {
  return new SqliteSaver(db);
}

export function createMemoryCheckpointSaver(): BaseCheckpointSaver {
  return new MemorySaver();
}
```

From 157 lines to ~10 lines. The entire `DrizzleCheckpointSaver` class is
deleted.

### 5.2 execution-trace-service.ts

```typescript
export interface ExecutionTraceService {
  getTrace(threadId: string): Promise<ExecutionTrace | null>;
  getStateAt(threadId: string, checkpointId: string): Promise<Record<string, unknown> | null>;
  listThreads(companyId: string, opts?: { limit?: number; status?: string }): Promise<GraphThreadRow[]>;
}

export class ExecutionTraceServiceImpl implements ExecutionTraceService {
  constructor(
    private repos: RuntimeRepositories,
    private checkpointSaver: BaseCheckpointSaver,
  ) {}

  async getStateAt(threadId: string, checkpointId: string): Promise<Record<string, unknown> | null> {
    const tuple = await this.checkpointSaver.getTuple({
      configurable: { thread_id: threadId, checkpoint_id: checkpointId },
    });
    if (!tuple) return null;
    return tuple.checkpoint.channel_values as Record<string, unknown>;
  }

  // getTrace() and listThreads() unchanged
}
```

## 10. Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| SqliteSaver internal table schema conflicts with our tables | Low | Table names differ (`checkpoints` vs `graph_checkpoints`) |
| `better-sqlite3` version mismatch between core and SqliteSaver | Medium | Pin compatible versions; test in CI |
| `channel_values` structure changes between LangGraph versions | Low | Pin `@langchain/langgraph` version; integration test catches this |
| SqliteSaver `setup()` call required before first use | Medium | Verify in integration test; document in factory |

## 11. Success Criteria

Phase 2.2 is complete when:

1. `DrizzleCheckpointSaver` class is fully removed
2. `createCheckpointSaver(db)` returns a working `SqliteSaver`
3. Graph execution writes checkpoints to SQLite (verified by E2E test)
4. Checkpoints survive simulated process restart (verified by E2E test)
5. `getStateAt(threadId, checkpointId)` returns real channel state
6. `meeting-flow` test verifies `llm_calls` recording
7. All tests pass (target: 95+)
8. Build passes for all packages
9. Design doc and implementation plan committed
