## Current 0.80.7 adoption outcome

The separate upgrade retained the high-level `pi-coding-agent` integration and
added only thin desktop projections:

- native PNG/JPEG/GIF/WebP bytes cross the execute and live-control contracts and
  enter Pi as `ImageContent`; parsed non-image files remain bounded prompt context;
- active-turn input uses Pi custom messages delivered as `steer` / `followUp`,
  while Pi owns its queue and emits `queue_update`; correlated control
  admission/consumption prevents false acknowledgement and duplicate retry;
- Pi compaction, automatic retry, settled, and context-usage events cross one
  neutral protocol-v8 `lifecycle` envelope for status display only;
- the headless `ExtensionUIContext` implements the four blocking requests
  `confirm`, `select`, `input`, and `editor` through the correlated
  FIFO `uiRequest`/answer channel;
- surviving root hosts are adopted before interrupted-run recovery, with the
  active UI request and accepted pending controls resurfaced after renderer
  reload rather than starting a duplicate session;
- root intent is admitted before the visible user message and host launch; final
  transcript/interaction persistence acknowledges Result/Error before the root
  terminal marker and retained-stream release, with replay bounded to 4,096
  events and 8 MiB while keeping the authoritative tail;
- the live host ledger preserves pending/accepted/failed/rejected outcomes and
  rejects conflicting reuse of a control id; only consumed controls cross a host
  restart through their Pi JSONL custom-message proof, while accepted but
  unconsumed work is retried from Offisim's durable visible queue;
- controller-owned Resume continues the same durable root, opens the recorded Pi
  JSONL exactly, or creates a fresh replay only when no session file was recorded;
  an invalid recorded path fails closed;
- root and child worktrees use Pi's canonical project-trust decision, while
  Enhance/collaboration use in-memory sessions with project resource discovery
  disabled.

Offisim still does **not** own a second tool loop, provider retry policy,
compaction algorithm, model/provider catalog, or complete Pi session tree. The
ordinary thread turns use `SessionManager.continueRecent(...)`; durable Resume
uses its recorded session file, or a fresh replay only when no file was recorded.
Pi owns JSONL history, branching, and deeper session navigation.

## Build / bundle coupling

The host is esbuild-bundled (`scripts/build-pi-agent-host.mjs` → `build-agent-host-lib.mjs`,
`bundle:true`, target node22) into `apps/desktop/src-tauri/resources/pi-agent-host.mjs`;
the Rust host spawns that frozen bundle (no runtime npm linkage) and validates
`PI_HOST_PROTOCOL_VERSION` on the `ready` handshake. So a pin bump is a build-time change:
update pin → rebundle → the Rust host loads the new bundle. No wire-protocol change is
implied by the SDK bump itself.

## Historical decision (2026-06-25)

- **NO-GO to bundle the 0.80.2 upgrade into Mission MVP.** Keep `0.79.8` pinned through
  M0–M2. Bundling an SDK bump with schema/UX/Mission work violates the loop hard rule and
  PRD §16.5/§16.6 and would muddy attribution if a regression appears.
- **GO as a dedicated, low-risk upgrade Epic** when scheduled. The spike de-risks it: the
  documented breaking changes don't touch Offisim's surface, so the expected work is
  mechanical (bump pin + verify typebox 1.1.38 still satisfies Pi 0.80.2's peer range +
  rebundle + run the full gate matrix). Do NOT trigger it before Mission MVP unless a
  security/critical fix lands only in 0.80.x.

## Current decision (2026-07-15)

- **Use exact `0.80.7`.** The manifest, lockfile, installed SDK, and npm `latest`
  agree on that version.
- Keep the high-level SDK seam. The current additions project Pi-native image,
  queue, lifecycle, and UI-request capabilities; they do not recreate Pi policy.
- Any later Pi upgrade remains a dedicated exact-pin change with registry and
  changelog verification, rebundle, harnesses, and release `.app` evidence.

## Historical upgrade Epic checklist (PRD §16.5)

This was the handoff checklist for the later upgrade. It is retained to explain
the original decision boundary; it is no longer the active pinning plan.

1. Lock the 0.79.8 behavioral baseline (capture `pnpm qa:loop` ledger + release `.app` evidence).
2. Bump pin to 0.80.2; refresh lockfile; confirm typebox peer range + node engines (`>=22.19.0`).
3. Verify the surface matrix above against 0.80.2 types (createAgentSession opts, registries, session events, `message.usage.cost.total`, extension `registerTool`/`tool_call`/`ctx.ui`, `bindExtensions`).
4. Rebundle `resources/pi-agent-host.mjs`; confirm `PI_HOST_PROTOCOL_VERSION` handshake still valid.
5. Run all Pi gates: `harness:pi-agent-host`, `check:pi-wire-contract`, `harness:pi-permission`, `cargo test`, `pnpm validate`.
6. Live `delegation` concurrency + `permission`/HITL (ask) + a real z.ai run via release `.app`.
7. Confirm existing on-disk Pi sessions still load (`SessionManager.continueRecent`).
8. Two consecutive clean `qa:loop` runs + release `.app` live scenarios twice.
9. Commit separately; do not mix with Mission schema/UX.
