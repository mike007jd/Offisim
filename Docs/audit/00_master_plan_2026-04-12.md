## Non-negotiable framing (LOCKED)

Downstream phases MUST NOT violate these. Any finding / suggestion / action that contradicts the framing is wrong. If you believe the framing is wrong, stop and ask the user before acting.

### F1. Core employee runtime is fixed and narrow

Offisim employees run exclusively on these LLM adapters:

- `packages/core/src/llm/anthropic-adapter.ts` (Anthropic SDK, BYO-key)
- `packages/core/src/llm/openai-adapter.ts` (OpenAI SDK, BYO-key)
- `packages/core/src/llm/subscription-adapter.ts` (ACP = Agent Client Protocol, `claude` CLI subprocess, desktop-only subscription mode)

**Do not** audit these as mockup/dead code candidates. **Do not** refactor them as part of any phase. They are the product's architectural load-bearing core. Issues with them are separate tickets outside this audit.

### F2. A2A and OpenClaw code are sleeping assets, not architecture

`packages/core/src/a2a/` and `packages/core/src/gateway/openclaw-client.ts` are **extension-point code for future external-agent interop**. They exist as hedges for agent runtimes that don't exist yet. They are NOT:

- architectural pillars
- product features
- deprecated code
- core runtime

They are code assets in cold storage. Phase 5 leaves them untouched. Phase 6 is a roadmap for *eventually* activating them (via `@a2a-js/sdk` replacement), but Phase 6 is not part of 1.0-rc.1 ship.

**Do not** add `@deprecated` tags. **Do not** write architecture docs about them. **Do not** add CLAUDE.md sections. **Do not** promote them to first-class citizens in any way. **Maximum permitted visibility** in CLAUDE.md is one terse line in the Gotchas section.

### F3. What counts as ship blocker

Ship blocker = user-facing UI that pretends to offer a working feature while the backing logic is fake/mock/placeholder. Silent backend code that no UI reaches and no runtime instantiates is NOT a ship blocker, no matter how unfinished it is.

### F4. Scope discipline

This audit is for 1.0-rc.1 ship readiness. It is NOT:

- bundle size optimization
- repository dedup
- test coverage expansion
- Scene V2 refactor
- the A2A 2.0 roadmap work itself (Phase 6 is planning only)
- any known-debt item listed in MEMORY.md Known Debt section

Adding new work beyond what findings justify is scope creep and must be refused.

### F5. What Phase 5 is allowed to change

Phase 5 makes small, surgical, reversible changes. No new features. No refactors beyond what findings require. No architecture documents (except the one-line CLAUDE.md addendum specified in Phase 2). Execution goes through ordered commits, each tied to a specific finding ID. Auto-pause every 5 commits for user status report.

## Phase 6 summary (for quick reference)

Full plan in `06_a2a_interop_plan_2026-04-12.md`. Key points:

- **Not a 1.0 phase.** This is the 2.0 roadmap for activating the A2A sleeping asset.
- **Adopt `@a2a-js/sdk@0.3.13`** (official Linux Foundation–governed TypeScript SDK, 1 runtime dep, Apache-2.0). Delete Offisim's ~548 lines of handwritten A2A code; replace with ~300 lines of SDK wiring + new Settings UI + Tauri sidecar.
- **Tauri sidecar pattern**: reuse the `subscription-adapter.ts` ACP subprocess pattern — run a Node subprocess hosting `express` + `@a2a-js/sdk/server/express`, bind `127.0.0.1` only, provisioned Bearer token.
- **Opt-in feature**: Settings "A2A Peers" tab, sidecar only starts when user enables.
- **Net code delta**: negative (~250 line reduction).
- **Estimate**: 2-3 weeks focused dev.
- **15 tasks** (6.1 through 6.15) with explicit dependencies and out-of-scope list.
- **Decision points deferred to 2.0 start**: MCP server mode parallel track? Sidecar cost budget? `OpenClawClient` final fate? Public agent card publishing?

Phase 6 is **roadmap planning**, not implementation. No code changes land from Phase 6.
