## HIGH

### R1. `company-template-service.ts` transact() rollback gap on async repo errors
**Severity: HIGH (happy path OK; failure path silently partial-commits)**

- File: `packages/core/src/services/company-template-service.ts:340-427` (drizzle path, `this.transact` provided)
- Pattern: `this.transact(() => { for (...) { void this.employeeRepo.create(...); } ... void this.sopTemplateRepo.create(...); void this.officeLayoutRepo.create(...); void this.zoneService?.seedSystemZones(...); for (...) { void this.prefabRepo.create(...); } })`
- Underlying repo: `packages/core/src/runtime/drizzle-repositories.ts:314-326` — `async create(emp) { ... db.insert(...).values(...).run(); return { employee_id }; }`. The `async` marker is cosmetic; the body is pure synchronous `better-sqlite3` with zero `await`.

**Why happy path works**: because the repo body runs synchronously up to `return`, the actual SQLite write commits inside the same tick as `transact()`. The only async thing is JS wrapping the returned value in `Promise.resolve(...)`, discarded by `void`. So 99% of the time, company materialization works.

**Why failure path is broken**: any sync `throw` inside an `async` function body becomes a **rejected Promise**, not a sync throw. If `db.insert(...).run()` raises (e.g. unique-key collision on pre-generated employee_id, disk full, FK violation from a corrupted template), the `async create()` wrapper captures the throw, returns a rejected promise, `void` discards it → the `transact()` callback sees no sync throw → better-sqlite3 commits the transaction anyway.

**User-visible failure mode**: Partial company materialization — e.g. the 1st employee + 2 zones committed, the 3rd employee constraint-violates, the 4th employee + layout + prefabs never run (because the callback aborted via rejected promise swallowed by void). The company loads with a broken roster, no error shown to the user.

Note: this is also why the CLAUDE.md gotcha ("Employee repo create() 可选 employee_id, transact() 中必须用预生成 ID (非 void promise.then())") exists — it addresses the ID determinism side of the same pattern but not the error-propagation gap.

**Fix direction (Phase 5)**: smallest surgical fix is to drop the `async` keyword from all `create(...)` methods in `drizzle-repositories.ts` that have no `await` in the body. They continue to satisfy the `Promise<T>` interface via TS's structural typing (a sync function returning `T` is assignable to a function returning `Promise<T>` only if the type is written `Promise<T>` — needs verification), OR wrap with `Promise.resolve` explicitly, OR collect rejections at the call site and rethrow inside `transact`. Prefer the last approach: inside `transact`, wrap each call in `.catch((e) => { throw e; })` or use a tiny helper that synchronously awaits and rethrows. This keeps the repo interface stable and makes the transaction actually atomic.

**Verification**: parity test at `apps/web/src/__tests__/unit/repository-parity.test.ts` governs the 3-way dedup; any interface change needs to pass it. If the fix drops `async` from drizzle, the memory + tauri impls must either match or declare a looser superset.

## MEDIUM

### R2. `useProjectAssignments.ts` `.then()` without `.catch()` — silent empty on repo error
**Severity: MEDIUM (feature silently shows empty on fetch failure)**

- File: `packages/ui-office/src/hooks/useProjectAssignments.ts:25-29`
- Code: `void repos.projectAssignments.findByProject(projectId).then((nextAssignments) => { if (!cancelled) setAssignments(nextAssignments); });`
- No `.catch()`. If `findByProject` rejects (DB unavailable, schema mismatch, stale repos after reinit), the error is swallowed as an unhandled promise rejection and `assignments` stays at the previous empty state. Consumers see an empty assignments list with zero indication anything failed.

**Failure mode**: Project detail panel shows "no employees assigned" after a DB hiccup, indistinguishable from "truly empty project." No toast, no log, no retry affordance.

**Fix direction (Phase 5)**: add `.catch((err) => { if (!cancelled) { console.error('[useProjectAssignments]', err); /* optionally setError */ } })`. Follow the same pattern as other data hooks in `ui-office/src/hooks/` which consistently use mounted-flag + catch.

### R3. Employee editor numeric fallback snaps empty input to 4096 mid-edit
**Severity: MEDIUM (UX bug, not data corruption)**

- Files:
  - `packages/ui-office/src/components/employees/EmployeeEditorDialog.tsx:543` — `updateField('maxTokens', Number.parseInt(e.target.value, 10) || 4096)`
  - `packages/ui-office/src/components/employees/interview-steps/ModelStep.tsx:65` — same pattern
- Pattern problem: `parseInt('') → NaN → NaN || 4096 → 4096`. User editing "4096" who wants "8192" may backspace through the field, and the intermediate empty-string state is remapped to 4096, producing a visible flicker. Worse: user types "0" → `0 || 4096 → 4096` (the `min={1}` HTML attribute catches this only at form submit, not at onChange).

**Failure mode**: Subtle UX friction — when clearing + retyping a max-tokens value, the field snaps back to 4096 between keystrokes. Not a ship blocker but notable quality gap.

**Fix direction (Phase 5)**: replace `|| 4096` with explicit NaN handling: `const n = Number.parseInt(e.target.value, 10); updateField('maxTokens', Number.isFinite(n) && n > 0 ? n : 4096);`. Or use nullish coalescing carefully (`?? 4096` won't help because `NaN ?? 4096 → NaN`). Apply same treatment to both call sites.

## LOW

### R4. `ChatInput.tsx` textarea has no `maxLength` — oversize paste can stall render
**Severity: LOW (defensive gap)**

- File: `packages/ui-office/src/components/chat/ChatInput.tsx:357-366`
- Textarea accepts unbounded paste. No length check in `handleChange` (line 137) nor before `onSend` (line 192). Pasting a multi-MB blob triggers: `setText(val)` → React re-render of the full ChatInput tree → `resizeTextarea` reads scrollHeight → possible render-thread stall → the blob then flows to the LLM adapter, which will itself reject but only after significant work.

**Failure mode**: User pastes a giant log file → UI freeze for seconds → eventual LLM 400. Not a crash, but a poor-feedback path.

**Fix direction (Phase 5 or later)**: add `maxLength={8000}` (or similar) on the textarea element + a short toast / inline warning if user attempts to paste beyond it. 8000 chars comfortably exceeds any normal chat turn; LLM truncation limits apply further down.

## False positives (agent-reported, verified not-a-bug)

Recording so future audits don't re-discover these.

### FP1. "Temperature `|| 0` rejects user-entered 0.0"
- `EmployeeEditorDialog.tsx:526`: `Number.parseFloat(e.target.value) || 0`
- Agent flagged as R-input-2 MEDIUM. **Verdict: not a bug.** `parseFloat('') → NaN → NaN || 0 → 0`; `parseFloat('0') → 0 → 0 || 0 → 0`; `parseFloat('0.5') → 0.5 → 0.5 || 0 → 0.5`. The fallback coincides with the minimum valid value, so behavior is identical to `?? 0`. No UX regression.

### FP2. "BootstrapProvider has a half-initialized repos race window"
- Agent flagged `BootstrapProvider.tsx:38-80` at R-conc-2 MEDIUM. **Verdict: not a bug.** `runtime` state transitions atomically between `null` and fully-loaded `RuntimeBundle`. `createTauriRuntimeReposOnly` / `createBrowserRuntimeReposOnly` construct the entire bundle before returning; React's `setRuntime(nextRuntime)` is a single state update. The `value` memo rebuilds with `runtime?.repos ?? null`, so consumers observe either `null` or fully-ready. No partial-exposure window.

### FP3. "company-template-service transact writes never execute"
- Agent flagged R-conc-1 CRITICAL with the claim that async promises inside sync `transact()` never persist. **Verdict: partially wrong.** Writes DO execute on happy path because the repo `async create` body is pure-sync `better-sqlite3` (no `await`). The real bug is the **failure path rollback gap**, recorded as R1 HIGH — see above. Downgraded from CRITICAL to HIGH accordingly.
