> **状态（2026-07-22 复核）**：需求已实现；本文只保留设计依据，不再是
> 待审批或待执行任务。当前行为与验收以代码、`Docs/FEATURES.md` 和 retained

对标(checkedAt 2026-07-16 调研):Claude Code auto-memory、Jules 仓库偏好、
Cursor Memories——该能力正从差异化滑向标配。Offisim 的差异化外壳:记忆是
**员工的资历**,连着玩具化人设与公司叙事(员工越用越懂你的仓库,比稿胜负
也沉淀成经验)。

以下为已拍板的设计方向 + 理由;有异议的条目请直接批注,无异议即为开发口径。

## 1. 记什么(记忆分类)

| 类型 | 例子 | 来源 |
|---|---|---|
| 踩坑 | 「此仓库 build 必须先 shared-types 再 renderer」「测试库有只读 fixture 不能动」 | run 中真实失败→修复的教训 |
| 仓库偏好 | 「提交信息用 conventional commits」「样式一律走 grammar.css 语汇」 | 用户打回/批注、AGENTS.md 之外的隐性约定 |
| 惯用约定 | 「验证命令是 pnpm verify」「产出放 Docs/reports/」 | 反复出现的操作模式 |
| 复盘要点 | 「上次比稿落选因为没跑验证就交稿」 | run/比稿收尾提炼 |

**粒度 = 员工 × 项目**。第一版不做跨项目泛化(避免把 A 仓库的坑套到 B 仓库)。

**不记**:凭据/密钥、会话原文大段、用户个人信息、与项目无关的闲聊。
写入前过敏感信息过滤(复用现有脱敏正则族)。

## 2. 何时写入

- **run 收尾自动提炼**:每次任务(含失败的 run——踩坑价值更高)收尾时,
  由该员工用一次轻量提炼调用输出 0-3 条结构化候选;无新知识就零条,不硬凑。
- **比稿收尾**:落选方也提炼(「为什么输」),连通 W5。
- **玩家手动添加**:档案页可直接写一条(玩家教员工规矩)。
- **去重合并**:新条目与既有条目语义冲突时更新原条目(updated_at 刷新),
  不无限追加;每员工每项目上限 30 条,满了按「最久未命中」淘汰。

## 3. 存哪

**本地 DB 表为唯一真相**(baseline schema 加表 + bump LOCAL_SCHEMA_VERSION,
prelaunch 不写迁移)。字段:员工/项目/类型/内容/来源 run/created/updated/
置顶标记/命中计数。

理由:结构化条目要支撑档案页编辑删除、注入排序、上限淘汰,DB 直接了当;
不放 vault 文件(vault 语义是技能与知识库,混入运行时记忆会污染来源;
Claude Code 的文件式记忆服务的是「用户可直接改文件」,Offisim 玩家改记忆
走档案页 UI,同一诉求已被满足)。

## 4. 如何注入

- **Pi 员工**:persona 组装缝(`buildEmployeeSystemPrompt` 旁)新增
  「项目经验」节:取该员工×该项目 top-N 条(置顶优先→命中数→最近更新),
  硬预算上限(≈1.5k tokens),超限截断。
- **外部引擎员工(Codex/Claude CLI)**:同一批条目经 W7 的 overlay 机制
  以只读上下文文件带进会话(与技能同路),换引擎不丢经验。
- 注入即记「命中」:被注入的条目 hit_count+1,反哺排序与淘汰。

## 5. 档案页「资历」呈现

Personnel 员工档案新增「经验」区:

- 按项目分组的记忆列表:类型徽标 + 内容 + 「来自:某次任务」溯源链接;
  可编辑、删除、置顶;可手动新增。
- 顶部资历摘要(全部由既有数据推导,不新增记账):完成任务数、比稿战绩
  (胜/参与)、活跃项目数。
- 全程人话,零 UUID(沿用 W1 返工确立的标准)。

## 6. 边界(第一版明确不做)

- 不做跨员工共享记忆(公司级知识沉淀归 Vault 知识库,另一条产品线)。
- 不做记忆自动泛化/嵌入检索;top-N 排序注入足够第一版。
- 不做「员工性格随记忆演化」(玩具化人设联动留后续)。
- 不做导入导出(prelaunch)。

## 7. 验收场景(live)

1. 员工在项目 A 踩坑(如用错验证命令被打回)→ run 收尾档案页出现对应踩坑
   条目 → 同项目再派一单,员工不再犯(会话可见其引用了该经验)。
2. 玩家在档案页手写一条仓库偏好 → 下一单员工遵循;删除该条 → 再下一单
   不再遵循。
3. Codex 引擎员工同样带经验开工(overlay 文件可见 + 行为遵循)。
4. 比稿落选方沉淀「为什么输」条目,档案页可见且溯源到该次比稿。
5. 上限淘汰与去重合并各构造一例,DB 与 UI 一致。
