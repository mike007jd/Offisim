## Outcome

Close every **verified** defect from the GPT-5.5 static audit, in priority order,
with maker/checker separation and evidence-backed gates, leaving `main` clean and
release-verified. Non-issues and over-stated claims are explicitly *not* worked.

**User-visible success demo (release `.app`):**
- Delete a company → no residual `attachments/<companyId>` blobs on disk.
- `git clone file:///etc/...` (or any out-of-workspace source) from the agent skill
  path is **rejected**, not copied into the sandbox.
- A registration using the official-seed email cannot adopt the seed account.
- Activity "All time" pages past the 200-row-per-table wall.
- `pnpm validate` + `cargo test` + new targeted tests green on the merged revision.

## Scope / Non-goals / Protected behavior

**In scope:** the 28 CONFIRMED + 5 re-scoped PARTIAL findings below.

**Explicitly OUT of scope (verified non-issues — do NOT touch):**
- **PE2** createVersion version-number race — `createVersion` has **zero callers**
  (dead code); the auditor's "rollback is correct because in-tx" rationale misreads
  the Tauri isolation model (correctness is the write-mutex, not tx placement).
  At most a dead-code cleanup, never a race fix.
- **D1** ZIP/OOXML/PDF boundaries — verified solid; nothing to fix.
- Full "unified SecretStore / system keychain" rewrite — see decision D-1; this loop
  does **bounded** secret hardening, not a keychain migration epic.
- "Eliminate double-repository drift via shared command builders" as a refactor — this
  loop installs a **contract-test oracle** (Wave 0) instead of a speculative rewrite.

**Protected behavior:** thin agent-agnostic GUI (CLAUDE.md), Pi harness/gateway path,
sandboxed `project_*` Tauri commands, OpenRouter/provider policy, no renderer-root
outer shell margin. Do not regress live-verified flows (delegation, permission modes,
reasoning streaming, attachments read).

## Repository facts

- Monorepo: `packages/{core,db-local,db-platform,doc-engine,install-core,registry-client,shared-types}`,
  `apps/{desktop,platform}`, renderer at `apps/desktop/renderer`.
- Two SQLite repo backends: **better-sqlite3** (`packages/core/.../drizzle.ts`, sync,
  rejects Promises in `transact`) and **sqlite-proxy** (`apps/desktop/renderer/src/lib/tauri-repos`,
  async, serialized via `withTauriSqlTransaction` write-mutex). Drift between them is the
  root of O1.
- Local writes go through the Tauri `tauri_plugin_sql` plugin with raw `db.execute`/`db.select`.
- Rust host lanes: `git.rs` (no timeout, no source check), `builtin_tools.rs` (hardened
  shell/file lane), `attachment_store.rs` (`attachments/` root, separate from `workspaces/`),
  `local_paths.rs`, `redaction.rs`, `mcp_bridge/*`.
- Platform: `apps/platform/src/middleware/auth.ts`, `services/moderation.ts`,
  `services/validation.ts`, `services/publish.ts`.
- Gates available: `pnpm validate` (turbo typecheck ×packages + harness + biome + knip
  deadcode), `cargo test`, GitNexus `impact`/`detect_changes`, release `.app` at
  `apps/desktop/src-tauri/target/release/bundle/macos/Offisim.app`.

## Human-owned decisions (authority boundary)

- **D-1 (secrets):** This loop does **bounded** secret hardening (Wave 3-S): encrypt
  A2A token + marketplace token at rest behind an opaque local key, and document the
  renderer-raw-SQL risk; it does **not** rip out `sql:allow-execute` (the entire renderer
  data layer depends on it) nor migrate to OS keychain. A full SecretStore/keychain epic
  is a separate proposal. *Rationale: removing raw SQL = rewrite the whole repo layer;
  pre-launch ROI says reduce plaintext + least-privilege the fs scope first.*
- **D-2 (S4 fs scope):** Narrow `fs:allow-app-*-recursive` to the specific subpaths the
  renderer actually needs (db file + its dir), removing blanket recursive app-data access,
  IF live verification shows nothing else depends on the broad scope. Else: document and
  defer. *Reversible; verify before committing.*
- **D-3 (PE1):** Either record an employee version on normal Profile/Appearance saves, **or**
  delete the unused `EmployeeVersionService`/History-fixtures and make History honest.
  Default: **wire versions on save** (History is a shipped surface). Escalate if it
  conflicts with the thin-GUI rule.
- **D-4 (ST4):** Either add a custom-prefab render path + install-time builtin-id
  validation, **or** block prefab/office_layout package install with a clear error until
  the render path exists. Default: **block install with a clear error** (cheaper, honest;
  prefab packages are not a shipped path today).
- **D-5 (push/deploy):** Merge to **local `main`** only when authorized. **No push, no
  remote merge, no platform deploy** without explicit user go-ahead. Platform DB changes
  (PL2/PL3) need a migration review before any deploy.

These are the only items to surface; everything else the lead decides and executes.

## Review lanes & adjudication

- Every change → a **fresh** reviewer agent inspecting the diff for: correctness, the
  named acceptance criterion, scope discipline (no opportunistic refactor), and the
  CLAUDE.md hard rules (thin GUI, no shell margin, provider policy).
- Security-sensitive items (G2, S4, PL2, AC2) → a second adversarial reviewer tasked to
  **bypass** the fix.
- Adjudicate with deterministic evidence (the oracle test) over opinion.

## Per-task gates (CLAUDE.md hard rules)

1. **Before editing any symbol:** `impact({target, direction:"upstream"})`; report blast
   radius; warn on HIGH/CRITICAL.
2. **Before commit:** `detect_changes({scope:"compare", base_ref:"main"})` — changes only
   touch expected symbols/flows.
3. Maker ≠ checker. Evidence (test/build/live screenshot) before any "done".
4. Release verification only via the real release `.app` path (no dev webview).

## Merge path & post-merge verification

- Integrate by wave into local `main` (per repo convention: commit, do not push).
- After each wave: `pnpm validate` + relevant `cargo test` on the integrated revision.
- After all waves: full matrix + release `.app` live demo of the success criteria above.
- Platform changes (PL1/PL2/PL3, M2) reviewed for migration safety; **no deploy** w/o D-5.

## Stop / escalate / cleanup

- **Escalate:** D-1..D-5 if context can't resolve; any platform DB migration; if a fix
  forces a thin-GUI violation; if S4 fs-narrowing breaks a real dependency.
- **Stop a task:** if the oracle can't be made deterministic — fall back to fresh-context
  review + reproducible manual demo, and say so.
- **Cleanup (definition of done):** every verifier/worker consumed then closed; worktrees
  archived/removed after integration; no debug logging or temp scaffolding left; `main`
  clean; final report lists what was fixed, what was proven a non-issue, and what was
  deferred (C4/C5, full SecretStore epic, A1/A2 if accepted).

## Suggested execution order

1. **Wave 0** (oracles) — unblocks everything; contract suite must fail on `setActive` first.
2. **Wave 1** (P1 + platform/git cheap wins) — release blockers.
3. **Wave 2** (consistency/fidelity) — parallel across disjoint boundaries.
4. **Wave 3** (product contract/quality + decisions D-3/D-4) — `setActive` (O1) last in the
